# ADR-0003: Структура репозитория и стратегия доноров

**Статус:** Accepted
**Дата:** 8 июня 2026 г.

---

## Контекст

ERA XDR — мультиязычная (Rust + Go + Python) платформа из ~12 модулей с общим
контрактом события. Нужно решить: монорепо или полирепо, и как именно использовать
open-source доноров (форк / обёртка / переписывание).

## Решение 1: Монорепо с жёсткими границами модулей

| Критерий | Почему монорепо |
|---|---|
| **Contract-first** | Изменение `Envelope`/proto и всех потребителей — одним атомарным PR. Невозможно случайно рассинхронизировать версии. |
| **Кросс-язык (Rust+Go)** | Один источник `.proto` → codegen в Rust и Go из одного места (связка с Apicurio). |
| **Atomic refactoring** | Меняешь схему — сразу видишь и чинишь всех потребителей. |
| **On-prem поставка** | Заказчику отдаётся согласованный срез всей системы одной версией, а не пазл из 12 репозиториев. |

### Целевая структура

```
era-xdr/
├── Cargo.toml               # Rust workspace (members = crates/*)
├── proto/era/v1/        # единый контракт (source of truth): envelope, ingest
├── crates/                  # Rust (Data Plane)
│   └── era-agent/           # [скелет] лёгкий сенсор конечной точки
│       └── (era-collectors — следующий)
├── services/                # Go
│   ├── ingest-gateway/      # [скелет] приём телеметрии
│   └── vm/                  # сканер уязвимостей (перенесён из /vm)
├── deploy/                  # dev-compose, ClickHouse DDL, (Helm/Terraform — далее)
│   ├── docker-compose.dev.yml
│   └── clickhouse/          # 001_schema.sql, 002_storage_policy.xml, 003_rollups.sql
├── docs/                    # ADR + MVP Sprint-1 spec
│   └── adr/
└── reports/                 # аналитика, blueprint
```

Будущие модули (`control-plane`, `detection-engine`, `ai-core`, `ui`) добавляются в
`services/`, `ai-core/`, `ui/` по мере фаз.

**Статус миграции:** ✅ выполнено. Модуль `vm` перенесён из `/vm` в `services/vm/`,
Go-неймспейс унифицирован: `era-itib/vm` → `era/services/vm`. Проект
переименован `era-itib` → **`era-xdr`**.

**Когда оправдан вынос в отдельный репо:** модуль становится самостоятельным
продуктом с независимым релиз-циклом и отдельной командой. Главный кандидат на
будущий вынос — `era-agent` (своя политика безопасности, подпись бинарей). Это
оптимизация будущего, не старта.

## Решение 2: Доноров переписываем с нуля (AI-Driven Reverse Engineering)

Мы **не форкаем** и **не оборачиваем** чужой код. Извлекаем паттерны/модели/
алгоритмы и реализуем заново на нашем стеке. Но «с нуля» применяется по-разному:

| Что берём | Как | Пример |
|---|---|---|
| **Код движка** | Переписываем с нуля (Rust/Go). Идеи, не строки. | Backpressure Vector → своя реализация на Rust |
| **Декларативные артефакты** | Импортируем/адаптируем (это данные/правила) | Sigma-правила, Nuclei-шаблоны, OCSF-схемы, MITRE |
| **Модели данных/протоколы** | Реимплементируем по спецификации | Prometheus exposition format, OCSF классы |

### Юридическая дисциплина по лицензиям доноров

| Лицензия | Доноры | Правило |
|---|---|---|
| **MIT / Apache 2.0** | Nuclei, Coraza, Prometheus, Fleet, Matano, Cilium, Teleport | Дружелюбны; технически можно переиспользовать код, но **переписываем ради единства стека** и контроля качества/безопасности |
| **MPL 2.0** | Vector | **Обязательно** реализовать заново; копировать код нельзя (copyleft на уровне файлов). Только идеи из документации/архитектуры |
| **Корпуса правил** | community Sigma/Nuclei templates | Лицензируются отдельно; **скан лицензий корпуса** перед импортом |

> Поскольку весь код доноров пишется заново, copyleft-загрязнения нет — держать всё
> в одном монорепо безопасно. Если бы линковали MPL-код Vector, пришлось бы
> изолировать его в отдельный процесс/модуль. Мы этого не делаем.

### Почему «с нуля» — преимущество, а не риск

Один согласованный лёгкий стек (Rust+Go) вместо «зоопарка» из 12 чужих рантаймов с
разными зависимостями, CVE и моделями памяти. Для air-gap on-prem поставки и
требований «лёгкости/надёжности» это решающий фактор: один контракт, один формат
логов, один способ деплоя.

## Последствия

**Плюсы:** атомарность изменений контракта; единый стек; чистая лицензионная
картина; простая on-prem поставка.

**Минусы / обязательства:** дисциплина границ модулей внутри монорепо; CI должен
собирать мультиязычный код; golden-тесты эквивалентности донорам обязательны.
