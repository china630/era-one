# ADR-0010: Лицензирование и офлайн-активация

**Статус:** Accepted
**Дата:** 8 июня 2026 г.
**Артефакты:** [`services/license/`](../../services/license/) (ядро + CLI `era-keygen`)

---

## Контекст

ERA XDR разворачивается в **air-gapped** контуре заказчика. Нужна защита бизнеса:
активация продукта по ключу с ограниченным сроком (1 или 3 года) и **помодульно**
(модель изданий, ADR-0005). Онлайн-активация невозможна — нет связи с интернетом.

## Решение

### 1. Криптомодель — офлайн-подпись (Ed25519)

- **Вендор** хранит приватный ключ (в идеале в HSM/KMS) и **подписывает** лицензию.
- **Продукт** несёт **встроенный публичный ключ** и проверяет подпись **локально**,
  без сети. Подделка без приватного ключа невозможна.
- Ed25519: компактные ключи (32/64 байта) и подписи (64 байта), быстрая проверка,
  современный стандарт. Реализация — Go stdlib `crypto/ed25519` (без внешних зависимостей).

### 2. Формат токена лицензии

```
ERA1.<base64url(claims_json)>.<base64url(signature)>
```

`claims_json` (подписанная полезная нагрузка):

| Поле | Назначение |
|---|---|
| `v` | версия формата |
| `lid` | license_id (ULID) — для учёта и отзыва |
| `cust` | заказчик |
| `tenant` | tenant_id |
| `edition` | базовое издание (`core`) |
| `modules` | включённые модули: `vm`, `ai`, `response`, `federated`, `national` |
| `max_nodes` | лимит узлов (до 150000) |
| `deployment` | отпечаток развёртывания (привязка, см. §3) |
| `iat` / `nbf` / `exp` | выпуск / начало / истечение (unix) |
| `grace_days` | grace-период после `exp` |

### 3. Привязка к развёртыванию (anti-copy)

Лицензия привязывается к `deployment` — стабильному ID инсталляции:
1. control-plane при первом запуске генерирует deployment ID и показывает заказчику.
2. Заказчик передаёт ID вендору (вне контура — email/портал).
3. Вендор выпускает лицензию, **привязанную** к этому ID.
Скопированная в другой контур лицензия не пройдёт проверку (`deployment` не совпадёт).

Для удобства dev/PoC привязка опциональна (пустой `deployment` = не привязана).

### 4. Поведение при истечении (НЕ жёсткий стоп)

Глушить security-продукт по таймеру опасно (оставляет заказчика без защиты).
Состояния (`Evaluate`):

| Статус | Условие | Поведение продукта |
|---|---|---|
| `VALID` | now ∈ [nbf, exp] | Полная работа |
| `GRACE` | exp < now ≤ exp + grace_days | Работает + настойчивые предупреждения |
| `EXPIRED` | now > exp + grace_days | **Деградация**: стоп приёма новых узлов/обновлений правил; ядро детекции продолжает работать (read-only режим) |
| `NOT_YET_VALID` | now < nbf | Не активирована |
| `NODE_LIMIT_EXCEEDED` | active_nodes > max_nodes | Блок онбординга новых узлов, алерт |

Точное поведение деградации — конфигурируемо политикой (банк ≠ госструктура).

### 5. Помодульная активация

`modules` в лицензии — источник истины для feature-flags изданий (ADR-0005).
control-plane включает модуль, только если он присутствует в валидной лицензии:
`HasModule("control-ai")` → активировать ERA Control AI (legacy alias `"ai"` поддерживается).

### 6. Отзыв (revocation) — CRL

Список отзыва — подписанный вендором документ со списком отозванных `lid`:

```
ERACRL1.<base64url(crl_json)>.<base64url(signature)>
crl_json = { "v":1, "iat":<unix>, "revoked":["lic-…","lic-…"] }
```

- **Подключённые контуры** — CRL подтягивается с сервера обновлений автоматически.
- **Air-gap** — CRL приходит на носителе вместе с обновлениями правил (раз в N недель).
  Продукт: `if lid ∈ CRL → деградация`.
- **Ограничение:** в air-gap отзыв действует только после следующего обновления;
  поэтому основная защита — **срок** (1/3 года), CRL — для подключённых и аккуратного
  управления.
- Компрометация приватного ключа → ротация ключа вендора + перевыпуск (поэтому
  публичный ключ — конфигурируемый параметр сборки, не «вечная константа»).

### 8. Защита от отката системных часов (anti-rollback)

В air-gap нет доверенного источника времени (NTP), поэтому заказчик технически
может **отмотать дату назад** и продлить истёкшую лицензию. Полностью предотвратить
нельзя, но обход делается дорогим и заметным через **sealed monotonic clock**:

1. **High-water mark.** Продукт хранит максимальное достоверное время в запечатанном
   (HMAC) виде и не позволяет времени «идти назад». Источники продвижения:
   - дата выпуска лицензии (`iat`) и подписанные даты в бандлах обновлений правил;
   - максимальные временные метки в потоке телеметрии (массово отмотать тысячи
     хостов нереально);
   - timestamps партиций ClickHouse.
2. **Эффективное время** = `max(local_clock, high_water)`. Если локальные часы
   оказались раньше high-water (с поправкой на skew) → **детект отката** → деградация.
3. **Откат часов = security-инцидент.** ERA XDR — XDR: откат системного времени на
   хосте детектируется как MITRE **T1070.006 (Timestomp/Indicator Removal)**. Попытка
   обойти лицензию датой **сама поднимает алерт**.
4. **Запечатывание** high-water — HMAC с install-secret (в проде — TPM/OS keystore).

Реализация: `SealedClock` (`EffectiveNow`, `Observe`) в `services/license`.
Для прода high-water mark **обязателен**; локальное время без печати — только dev.

### 9. Deployment fingerprint (детали привязки)

`deployment` вычисляется самим продуктом из стабильных признаков инсталляции:

```
deployment = SHA256(install_salt | machine-id | board-serial | disk-serials | MACs)
```

- control-plane показывает ID при первой установке → заказчик передаёт вендору →
  вендор привязывает лицензию. Продукт пересчитывает и сверяет при старте.
- **Толерантность N-из-M:** чтобы лицензия не «слетала» при штатной замене
  диска/сетевой карты, поддерживается частичное совпадение признаков
  (`MatchTolerant`). Строгий ID — для привязки, толерантное сопоставление — для
  устойчивости. Это ответственность control-plane (сбор признаков платформозависим).

### 10. Rust-верификатор агента (defense-in-depth)

Логика `Verify`/`Evaluate` зеркалируется в Rust-крейт `crates/era-license`
(`ed25519-dalek`), публичный ключ встраивается в агент при сборке. Формат токена
единый — Go и Rust читают одни лицензии. **Это дополнительный слой**, не основной:
клиентский бинарник теоретически патчится, поэтому первичный гейт — серверный
(control-plane).

### 7. Где живёт проверка

- **Вендор-сторона:** CLI `era-keygen` (генерация ключей, выпуск, проверка лицензий).
  Приватный ключ — только у вендора.
- **Продукт-сторона:** пакет `license` импортируется `control-plane`, который
  гейтит активацию модулей. (Rust-верификатор для агента — позже, при необходимости;
  Ed25519 есть и в Rust.)

## Последствия

**Плюсы:** надёжная офлайн-защита бизнеса; помодульная монетизация в одной модели с
изданиями; нет онлайн-зависимости; безопасная деградация вместо опасного стопа.

**Минусы / обязательства:** защита приватного ключа (HSM/KMS) — критична;
отзыв в air-gap ограничен (митигируется сроком); привязка к deployment требует
процесса обмена ID с заказчиком.
