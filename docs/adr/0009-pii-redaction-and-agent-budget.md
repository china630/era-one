# ADR-0009: PII-редакция и бюджет ресурсов агента

**Статус:** Accepted
**Дата:** 8 июня 2026 г.
**Контекст:** Два P0-принципа из ADR-0006, ранее не специфицированные:
очистка PII до записи в Data Lake и измеримая «лёгкость клиента».

---

## Часть A. PII-редакция (на агенте, до отправки)

### Принцип

PII маскируется/удаляется **на агенте, ДО** сериализации в `Envelope` и отправки.
В Data Lake (ClickHouse) персональные данные в открытом виде не попадают.
Флаг `pii_sanitized=true` проставляется только после прохождения редактора.

### Классы данных и действия

| Класс | Примеры полей | Действие по умолчанию |
|---|---|---|
| **Прямые идентификаторы** | имя пользователя ОС, email, ФИО | Псевдонимизация (HMAC-SHA256 с tenant-ключом) |
| **Сетевые идентификаторы** | приватные IP — оставляем; публичные с пользователем — оцениваем | Конфигурируемо по политике tenant |
| **Содержимое** | command line, тело файла, тело письма, query | Маскирование секретов по regex (токены, пароли, ключи, PAN) |
| **Аутентификация** | пароли, токены, ключи, PAN-карты | **Полное удаление** (никогда не покидает хост) |
| **Чувствительные пути** | домашние каталоги пользователей | Нормализация (`/home/<user>` → `/home/{uid}`) |

### Механизм псевдонимизации

- **HMAC-SHA256(value, tenant_pseudonym_key)** — детерминированно в пределах
  tenant (сохраняет корреляцию «тот же пользователь»), но необратимо без ключа.
- Ключ псевдонимизации хранится в KMS контура заказчика, агенту выдаётся
  control-plane'ом, ротация — по политике.
- Reverse-lookup (для легитимного расследования) — только через привилегированный
  сервис control-plane с аудитом (см. ADR-0006, самоаудит).

### Конфигурируемость

- Политики редакции — часть policy bundle (раздаётся через `Register`, ADR-0008).
- Регулятор/комплаенс может требовать иной уровень (банк ≠ госструктура) →
  политика per-tenant.
- Секрет-сканеры (regex для токенов/PAN/ключей) обновляются как контент,
  отдельно от кода агента.

### Тестирование

- **Golden-тесты редакции:** фикс. вход с PII → ожидаемый обезличенный выход.
- **Fuzzing** секрет-сканеров (защита от обхода).
- CI-gate: запрещён мёрж при провале golden-тестов редакции.

## Часть B. Бюджет ресурсов агента (измеримая «лёгкость»)

### Принцип

«Лёгкость клиента» — измеримый инвариант, а не лозунг. Нарушение бюджета = красный
CI и блок релиза агента.

### Целевые лимиты (steady-state, на типовом хосте)

| Ресурс | Лимит (target) | Hard-cap (kill/alert) |
|---|---|---|
| **CPU** | < 2% среднесуточно | 5% (троттлинг сбора) |
| **RAM (RSS)** | < 150 МБ | 250 МБ |
| **Диск (буфер)** | < 1 ГБ кольцевой буфер | 2 ГБ (drop oldest) |
| **Сеть (исходящая)** | < 50 МБ/час на хост (амортизировано) | burst-лимит на батч |
| **Размер бинарника** | < 30 МБ | 50 МБ |
| **Время старта** | < 2 сек | — |

> Значения — стартовые ориентиры Фазы 1; калибруются на нагрузочных тестах.
> Источник нагрузки (4.5 ТБ/сутки от 150k) → ~30 МБ/сутки на хост в среднем,
> что укладывается в сетевой бюджет.

### Самозащита через бюджет (связь с tamper protection)

- Агент **самоограничивается**: при превышении CPU/RAM включает троттлинг сбора
  (приоритет — детекция over полнота телеметрии), а не падает.
- Disk-buffer — кольцевой: при переполнении дропает самые старые НЕкритичные
  события, сохраняя детекции (`detection != null`).

### Enforcement в CI

- **Бенчмарки производительности** в пайплайне агента (criterion для Rust).
- **Нагрузочный профиль** (синтетический поток событий) → замер CPU/RAM/диск.
- Сборка **падает** при превышении target-лимитов; hard-cap — алерт + ручное ревью.
- Размер бинарника проверяется как артефактный gate.

## Последствия

**Плюсы:** приватность by-design (комплаенс-готовность); защита киллер-фичи
«лёгкий клиент» от деградации; предсказуемое поведение под нагрузкой.

**Минусы / обязательства:** поддержка KMS/ключей псевдонимизации в air-gap;
постоянные бенчмарки в CI; калибровка лимитов на реальном железе заказчиков.
