# ADR-0005: Независимость модулей и упаковка в издания (editions)

**Статус:** Accepted
**Дата:** 8 июня 2026 г.
**Контекст:** Стратегия апсейла — модули должны продаваться/разворачиваться поэтапно.

---

## Контекст

Бизнес-цель — апсейл: начать с базовой поставки и доращивать функциональность
модулями. Для этого модули должны быть слабо связаны (loosely coupled) и по
возможности разворачиваться/продаваться независимо.

## Механизм развязки

Развязка достигается двумя средствами, уже заложенными в архитектуру:

1. **Общий контракт `Envelope` + шина Kafka** — модули общаются через события, а не
   прямые вызовы. Любой модуль можно вынуть или продать отдельно.
2. **Topic-per-domain** — новый модуль просто подписывается на нужный topic, не
   трогая остальных.

## Карта автономности модулей

| Модуль | Автономность | Зависимость |
|---|---|---|
| `/vm` (сканер) | **Полная** | Работает сам по себе (уже работает) |
| `era-agent` + локальная Sigma | **Высокая** | Офлайн-детект; ценность растёт с ядром |
| `detection-engine` | Частичная | Нужен ingest + storage (фундамент) |
| `ai-core` | Надстройка | Над телеметрией |
| `soar` | Надстройка | Нужны детекции на вход |

«Автономен» и «без зависимостей» — разные вещи: одни модули продаются отдельно,
другие требуют фундамент (ingest + Kafka + storage).

## Решение: модель изданий (editions)

```
ERA Core (база)      → agent + ingest + storage + detection (Sigma) + dashboards
  └─ + ERA Vuln       (/vm — сканер уязвимостей)        ← апсейл
  └─ + ERA Control AI   (ai-core: SOC Analyst, hunting)   ← апсейл (главный)
  └─ + ERA Response   (soar: playbooks реагирования)    ← апсейл
  └─ + ERA Federated  (Фаза 3, federated learning)      ← апсейл (опция privacy)
  └─ + ERA National   (Фаза 4, нацхаб STIX/TAXII)       ← апсейл (опция гос)
```

| Издание | Содержит | Фаза | Тип |
|---|---|---|---|
| **ERA Core** | agent, ingest-gateway, control-plane, Kafka, ClickHouse, detection-engine, dashboards | MVP | База |
| **ERA Vuln** | `/vm` | в работе | Апсейл |
| **ERA Control AI** | `ai-core` (SOC Analyst, threat hunting, NL→SQL/Sigma) | 2 | Апсейл (главный) |
| **ERA Response** | `soar` | 2 | Апсейл |
| **ERA Federated** | federated learning внутри организации | 3 | **Опция (не по умолчанию)** |
| **ERA National** | межорг/нацхаб (STIX/TAXII) | 4 | **Опция (не по умолчанию)** |

> federated-издания (ERA Federated, ERA National) — **платные опции, не дефолт**
> (см. [`ADR-0002`](./0002-learning-topology.md)). Базовая поставка полностью
> автономна без обмена данными наружу контура.

## Требования к проектированию ради независимости

- Каждый модуль — отдельный деплой-юнит (контейнер) с feature-flag активацией.
- Никаких прямых синхронных вызовов между модулями analytics-плоскости — только
  через Kafka-события или явные gRPC-контракты control-плоскости.
- Лицензирование на уровне издания: активация модулей через лицензионный ключ
  (офлайн-подпись Ed25519, помодульно — см. [`ADR-0010`](./0010-licensing-and-activation.md)).
  Поле `modules` в лицензии — источник истины для feature-flags изданий.
- Деградация по graceful: при отсутствии `ai-core` система работает на Sigma-детекте.

## Последствия

**Плюсы:** гибкий апсейл; клиент платит за то, что использует; модули эволюционируют
независимо; чистая модель лицензирования.

**Минусы / обязательства:** дисциплина «через шину, а не напрямую»; контрактные
тесты на границах модулей; управление feature-flags и лицензионными ключами.
