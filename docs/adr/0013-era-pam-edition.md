# ADR-0013: Издание ERA PAM (Privileged Access Management)

**Статус:** Accepted (Implemented MVP; крипто-аудит HSM и RDP-review — [gate: external])
**Дата:** 27 июня 2026 г. · обновлено 1 июля 2026 г.
**Контекст:** Запросы заказчиков (пример: ManageEngine **Password Manager Pro** —
сейф на 200 администраторов, SSH/RDP-прокси) показывают потребность в корпоративном

## Контекст и проблема

PAM закрывает другую боль, чем XDR/Manage: **управление привилегированным доступом** —
где админы берут пароли от серверов, ключи SSH, и как контролируются их сессии.
Это серверный **vault + прокси**, а не агент на конечной точке.

Важно отделить от уже решаемого:
- **EPM-lite** (временная выдача прав локального админа на ноутбуке) — это **агент**,
  идёт в **ERA Manage** (§6 напр. 3, roadmap P6). **Не** PAM.
- **Запись привилегированных сессий** — частично **уже есть** в коде: `services/dlp`
  (`session.Store`: start/command/end + алерты). Это фундамент ERA PAM.

Чего нет и что нужно построить: **vault** (сейф секретов), **checkout** креденшелов,
**session-proxy** (SSH/RDP-шлюз с записью), ротация паролей.

## Решение (scope издания)

### 1. Компоненты ERA PAM (`services/pam`)

| Компонент | Назначение | Статус-отправная точка |
|---|---|---|
| **Vault (secret store)** | Хранение паролей/ключей/секретов админов, шифрование at-rest | новый |
| **Secret engines** | Статические секреты + (позже) динамические/ротация | новый |
| **Checkout / brokering** | Выдача креденшелов по регламенту (RBAC, approval, TTL) | новый |
| **Session proxy** | SSH/RDP-шлюз: доступ без раскрытия пароля, запись сессий | расширить `services/dlp` |
| **Session recording** | Журнал команд/экрана привилегированной сессии | **есть** (`services/dlp`) |
| **Audit / chain-of-custody** | Неизменяемый журнал доступа к секретам | переиспользовать `services/platform/custody` (hashchain) |

### 2. Криптомодель vault (инвариант безопасности)

- **Шифрование at-rest** секретов; мастер-ключ — **в HSM/KMS контура**, не в коде/конфиге
  (правило `security-and-air-gap`, ADR-0009). Никаких секретов в репозитории/логах.
- **Seal/unseal** хранилища (паттерн-донор HashiCorp Vault — **идея, не код**, ADR-0003):
  vault стартует «запечатанным», распечатывается ключами-долями (Shamir) операторов.
- **Zero-knowledge к UI:** портал не хранит и не логирует расшифрованные секреты.
- **Air-gap:** весь PAM — в контуре заказчика; никаких внешних KMS/SaaS.

### 3. Session proxy (SSH/RDP)

- Админ подключается к шлюзу ERA PAM, **не зная** целевого пароля (vault инжектит).
- Полная **запись сессии** (команды/видео) → события в `Envelope` (`domain=privileged`)
  → ingest → cases/SOAR. Подозрительная команда = detection (как сейчас в `dlp`).
- Донор архитектуры туннеля — **Guacamole/Teleport** (паттерн, не форк AGPL/BUSL-кода).

### 4. Интеграция с платформой

- **Лицензия:** новый модуль `pam` в claims (ADR-0010), feature-flag в control-plane
  (ADR-0005). Издание включается ключом.
- **Identity:** интеграция с AD/LDAP/SSO (как у portal) для аутентификации админов.
- **Multi-tenant:** `tenant_id` во всех записях vault/сессий (как в остальной платформе).
- **Аудит:** доступ к секретам и сессии — в hash-chain custody (неизменяемость для
  комплаенса банков/гос).

### 5. Что НЕ в MVP

- Динамическая ротация паролей на всех типах целей (поэтапно после статического vault).
- Полный обзор привилегий (PEDM enterprise) — отдельная линия.
- Облачные secret-брокеры (несовместимо с air-gap-приоритетом).

## Дорожная карта (ориентир)

Фаза **P4p** в [`ERA-Platform-Vision.md`](../ERA-Platform-Vision.md) §13:
код (AI-assisted) ~4–8 нед, **обязательный крипто-аудит vault/HSM** — отдельный гейт,
не ускоряется написанием кода. Старт — после exit пилота XDR.

## Последствия

**Плюсы:** закрывает PAM-строку запросов (Password Manager Pro-класс) в нашей
экосистеме; переиспользует session-recording (`services/dlp`), custody-hashchain,
лицензирование и identity; единый SOC-обзор (PAM-события в общий конвейер).

**Минусы / обязательства:** vault — высокорисковый компонент, **крипто-аудит и
управление мастер-ключом обязательны**; session-proxy (RDP) требует отдельного
security-review; ротация секретов — платформозависимая и трудоёмкая; нельзя
сокращать гейты ради сроков.

## Открытые вопросы (закрыты дефолтами плана Stage 8)

- **Свой vault** — да (не форк донора).
- **Мастер-ключ** — KMS/HSM-абстракция; dev = `software-sealed`.
- **dlp vs pam** — смежные сервисы; pam импортирует `session.Store`.
- **Запись** — команды/текст MVP; видео позже.
