# ADR-0014: Мульти-продуктовое монорепо (solo-профиль)

**Статус:** Accepted
**Дата:** 27 июня 2026 г.
**Контекст:** ERA выходит за рамки одного продукта (XDR) к нескольким изданиям
(XDR, Manage, PAM, Observe) при **одной команде (solo + AI)** и **едином релизе**.
Нужно зафиксировать, как развивать и хранить это в репозитории, не превращая
монорепо в спагетти. Расширяет [`ADR-0003`](0003-repository-structure-and-donor-strategy.md)
на multi-product этап.

**Связано:** [`ADR-0003`](0003-repository-structure-and-donor-strategy.md) ·
[`ADR-0005`](0005-module-independence-and-packaging.md) ·
[`ADR-0010`](0010-licensing-and-activation.md) ·
[`ERA-Platform-Vision.md`](../ERA-Platform-Vision.md) · `editions-control.yaml`

---

## Контекст и проблема

«Боль» масштабирования репо обычно не в выборе моно/поли, а в попытке раскладывать
дерево **по продуктам**. Но продукт (edition) — это **композиция** компонентов:
один и тот же компонент (агент, `proto`, `services/platform`, `services/dlp`) входит
в несколько изданий. Папки-по-продукту лишают общие компоненты «дома» и плодят
дублирование и перекрёстные зависимости.

Профиль решения — **solo + AI, единый релиз**: исключаем всё, что нужно только
большим командам (per-edition релиз-поезда, вынос в отдельные репо, CODEOWNERS),
оставляем минимум, который реально снижает когнитивную нагрузку и не даёт
AI-агентам разводить связи между модулями.

## Решение

### 1. Остаёмся в монорепо (подтверждение ADR-0003)

Единое git-дерево, единая версия всей платформы (`vX.Y`), один деплой с
feature-flags. Причины те же: contract-first, кросс-язык (Rust+Go), атомарный
рефактор, простая on-prem поставка одним срезом.

### 2. Две оси: физическая раскладка ≠ продукт

- **Физически** репо раскладывается **по слою/компоненту** (не по продукту).
- **Продукт (edition)** выражается **манифестом** `editions-control.yaml`, а не папкой.

Правило размещения нового кода — по природе компонента:

| Природа | Куда |
|---|---|
| Realtime/cron/on-demand на хосте | `crates/era-agent-core` или `crates/era-plugin-*` |
| Серверная логика (приём/детект/API/движок) | `services/<name>/` (свой `go.mod`) |
| Используется 2+ сервисами | `services/platform/` (shared kernel) — и нигде больше |
| Контракт события/API | `proto/` (только добавление, ADR-0001) |
| Правила/CVE/Sigma/IoC | `data/` |
| Сборка/деплой | `deploy/` |

### 3. Манифест изданий `editions-control.yaml` (single source of truth)

Связывает `edition → license_module (ADR-0010) → services → agent plugins →
deploy_profile → docs`. Один файл в корне (solo-профиль — без отдельного каталога).
Вход для `services/platform/licensegate` и `edition_matrix_test`. Статусы изданий:
`ga | ga-option | roadmap`; для не реализованных модулей лицензии — `exists: false`.

### 4. Go workspace (`go.work`) — атомарность без слияния модулей

Per-service `go.mod` **сохраняются** (независимость, `internal/`, `replace`), но
поверх добавляется корневой **`go.work`** со всеми модулями (`services/*`, `gen/go`,
утилиты). Даёт:
- атомарный кросс-сервисный рефактор при изменении `proto` (прямая поддержка
  contract-first из ADR-0003);
- единый `go build` по модульным путям, общий build-list;
- AI-агент видит весь граф сразу.

> Сборка в workspace — по **модульным путям** (`go build ./services/platform/...`),
> т.к. `./...` от корня (не модуль) и wildcard через границы модулей не работают.

### 5. Дисциплина зависимостей (проверяемый инвариант)

Граф импортов — только «вниз»:

- всё зависит от `proto`/`gen` (контракт), не наоборот;
- сервис зависит **только** от `services/platform` (shared kernel), **не** от другого
  сервиса напрямую — общение через Kafka/gRPC (ADR-0005);
- ни один сервис не импортирует чужой `internal/` (агрессивно используем `internal/`);
- плагины агента зависят от `era-agent-core` ABI + `proto`, не друг от друга.

**Enforcement (solo-профиль):** базовый барьер — Go `internal/` (работает уже сейчас,
бесплатно). Жёсткий CI-гейт `depguard` — конвенция «включить при появлении CI».
Готовый сниппет:

```yaml
# .golangci.yml (включить, когда заведёте golangci-lint в CI)
linters:
  enable: [depguard]
linters-settings:
  depguard:
    rules:
      services:
        list-mode: lax
        deny:
          - pkg: era/services
            desc: "сервис не импортирует другой сервис напрямую — только через platform/Kafka/gRPC"
        allow:
          - era/services/platform
          - era/contracts/gen
```

### 6. Что осознанно ОТЛОЖЕНО (до роста команды)

- ❌ Вынос в отдельные репо (включая `era-agent`) — оптимизация будущего (ADR-0003).
- ❌ Per-edition релиз-поезда и теги — единый релиз всей платформы.
- ❌ CODEOWNERS, micro-frontends — для нескольких команд.
- ❌ Схлопывание/миграция Go-модулей — `go.work` решает без переездов.
- ❌ Переименование `era-xdr` → `era-platform` — косметика, по желанию.

## Последствия

**Плюсы:** чистота монорепо без накладных расходов; добавление издания = запись в
`editions-control.yaml` + компоненты по слою, без переезда файлов; `go.work` снимает боль
синхронизации контракта; граница «сервис не лезет в сервис» проверяема.

**Минусы / обязательства:** `editions-control.yaml` нужно держать в актуальном состоянии
(привязать к `edition_matrix_test`); общий build-list в `go.work` унифицирует версии
зависимостей (обычно плюс, но следить за конфликтами); depguard — включить при CI.

## Артефакты

- `go.work` — корневой workspace (все Go-модули). Проверено: `go build` ключевых и
  остальных сервисов — PASS.
- `editions-control.yaml` — манифест изданий.
- Сниппет `.golangci.yml` (§5) — к включению при появлении CI.
