PRD (Product Requirements Document): Модуль /vm (Vulnerability Scanner)
Фаза: MVP (Minimum Viable Product)
Технологический стек: Go (Бэкенд), React/Next.js + Tailwind (Фронтенд внутри /core)
Донор бизнес-логики: Nuclei (ProjectDiscovery)

1. Product Scope (Границы MVP)
Что ВХОДИТ в MVP:

Парсинг и выполнение декларативных правил проверок (на базе YAML-синтаксиса Nuclei).

Сканирование заданного списка таргетов (IP-адреса, домены, поддомены).

Базовые типы сетевых проверок: HTTP (поиск открытых админок, старых версий CMS), TCP/DNS (открытые порты, устаревшие сертификаты).

Генерация структурированного JSON-отчета с результатами ("находками" / findings) и отправка его в модуль /core (в базу данных).

UI-виджет: Таблица с результатами сканирования и кнопка "Запустить скан".

Что НЕ ВХОДИТ в MVP (Оставляем на v2.0):

Автоматическое сканирование по расписанию (Cron-джобы).

Сложные многошаговые атаки (где шаг 2 зависит от результатов шага 1).

Автоматическая генерация тикетов в Jira/Slack при нахождении критической дыры.

2. Архитектура движка (Engine Architecture)
Движок пишется на Go с упором на максимальную параллельность (Goroutines).

Template Parser (Парсер шаблонов): ИИ-агенты должны переписать логику чтения YAML-файлов Nuclei. Парсер читает шаблон (например, check-nginx-version.yaml), понимает, какой HTTP-запрос нужно отправить и какое регулярное выражение искать в ответе.

Execution Engine (Исполнитель): Получает массив таргетов (например, 100 IP-адресов). Запускает пул воркеров (Go-рутин), которые асинхронно отправляют HTTP/TCP запросы к этим таргетам, применяя распарсенные правила.

Result Aggregator (Агрегатор): Собирает успешные срабатывания (уязвимости найдены) и отсеивает мусор (таймауты, закрытые порты).

3. API-Контракты (Интеграция с /core)
Микросервис /vm должен быть изолированным и общаться с интерфейсом только через строгие REST/gRPC контракты.

Входящий запрос на скан (Trigger Scan):

JSON
POST /api/v1/vm/scan
{
  "scan_id": "req-12345",
  "targets": ["192.168.1.0/24", "api.client-domain.com"],
  "severity_level": ["high", "critical"], 
  "concurrency": 50 
}
Исходящий результат (Findings Payload) — уходит в шину /core:

JSON
POST /api/v1/core/events/ingest
{
  "module": "vm",
  "scan_id": "req-12345",
  "timestamp": "2026-04-03T19:30:00Z",
  "finding": {
    "target": "api.client-domain.com",
    "vulnerability": "Exposed Git Repository",
    "severity": "critical",
    "description": "Found /.git/config file exposed to public internet.",
    "remediation": "Block access to hidden directories in web server config."
  }
}
4. UI/UX Требования (Для единого App-Shell)
Интерфейс модуля /vm интегрируется в общий сайдбар системы как раздел "Аудит уязвимостей".

Layout: Размещается в центральном контейнере (Dashboard). Используем корпоративную палитру (чистый белый фон, тени, акцентные цвета для статусов).

Секция Action: Форма ввода таргетов (текстовая область для вставки IP/доменов) и primary-кнопка "Запустить аудит".

Секция Results: Таблица thead/tbody с hover-эффектами.

Колонки: Дата, Цель (Target), Название уязвимости, Уровень угрозы (Severity).

Цветовое кодирование Severity: Critical — Красный badge, High — Оранжевый badge, Medium — Желтый badge, Info — Серый badge.

5. Задача для ИИ-агентов (Пайплайн рефакторинга)
Этап 1: Загрузить ИИ-агентам документацию Nuclei по структуре их YAML-шаблонов. Попросить написать на Go парсер, который будет переводить эти YAML в структуру struct языка Go.

Этап 2: Дать ИИ задачу написать HTTP/TCP Client Wrapper на Go, который будет брать параметры из этих struct и выполнять реальные сетевые запросы с заданным таймаутом.

Этап 3: Написать агрегатор ответов, который формирует итоговый JSON по нашему API-контракту.
